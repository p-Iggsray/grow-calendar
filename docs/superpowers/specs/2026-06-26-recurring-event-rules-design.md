# Recurring Event Rules — Design

Date: 2026-06-26
Status: Approved for planning

## Problem

Per-day tasks are generated by phase logic in `src/lib/growdata/detail.js` (`generateDetail`),
then optionally reshaped by two override layers:

- Phase overrides (`phase_overrides`): full-replace the task array for an entire phase.
- Day overrides (`plan_day_overrides`): per-day diff of edit / remove / append, keyed by `YYYY-MM-DD`.

There is no way to express a recurring, timed event such as "neem oil spray every 7 days from
transplant until flowering." Adding one today means hand-editing dozens of individual days or
rewriting whole phase task lists. The event also exists only as scattered copies, so changing or
removing it later means hunting down every affected day.

## Goal

Let the grower add a recurring/timed event once and have it appear correctly on every matching day,
remain a single editable object, and survive AI plan regeneration. Primary interface is MJ natural
language ("add a neem spray every 7 days during veg"); MJ writes to a structured, inspectable rule
store underneath.

## Decisions (from brainstorming)

- Support all four timing patterns: every N days in a range, anchored to a phase, relative to a
  milestone, and specific dates only. A single `{window × cadence}` model covers all four.
- Primary interface: MJ natural language, backed by a structured rule store. MJ gets tools to
  create, list, and delete rules.
- Visibility: task line only inside the day view. No month-calendar markers in v1.
- Skipping a single occurrence reuses the existing per-day remove mechanism (or "tell MJ to skip
  July 14"); rules are not edited for a one-day skip.

## Data model

New `event_rules` JSON array stored as a column on the `grows` table, sibling to `phase_overrides`,
`generated_plan`, and `survey`. The active grow's plan data lives in `grows`; the older
`plan_config` table is legacy and auto-migrates into `grows` on first `GET /api/grows`, so event
rules are stored and served per-grow from `grows`. NULL reads as an empty list.

Each rule:

```js
{
  id: "evt_a1b2c3",            // server-assigned, stable
  label: "Neem oil spray",     // for listing/management
  task: "Spray neem oil on leaf undersides to runoff", // the task line injected into the day
  enabled: true,
  window:  { ... },            // WHEN the rule is active
  cadence: { ... },            // WHICH days inside the window fire
  createdAt: "2026-06-26T12:00:00.000Z" // stable ordering key, server-assigned
}
```

`window` is exactly one of:

- `{ type: "range", from: "YYYY-MM-DD", to: "YYYY-MM-DD" }`
- `{ type: "phase", phases: ["veg_full", "pre_flower"] }` — matched via existing
  `getPhase(date, config)` membership; no phase-boundary math required.
- `{ type: "milestone", anchor: "transplant", offsetStart: 0, offsetEnd: 30 }` — resolves to a date
  range computed off a config date (`anchor` + offset days). `offsetStart === offsetEnd` means a
  single day. Offsets may be negative (e.g. `-3` = three days before the anchor).

`cadence` is exactly one of:

- `{ type: "everyDay" }`
- `{ type: "everyNDays", n: 7, anchor: "2026-05-24" }` — counts from `anchor`; if `anchor` is
  omitted, defaults to `config.start`. A date fires when `daysBetween(anchor, date) % n === 0`.
- `{ type: "weekdays", days: ["mon","thu"] }`
- `{ type: "dates", dates: ["2026-06-30","2026-07-14"] }` — `window` is optional for this cadence;
  the list itself is the source of truth.

### Pattern coverage

- Every N days in a range: `window=range` + `cadence=everyNDays`.
- Anchored to a phase: `window=phase` + `cadence=everyDay` / `weekdays` / `everyNDays`.
- Relative to a milestone: `window=milestone` (offsetStart=offsetEnd=-3) + `cadence=everyDay`.
- Specific dates only: `cadence=dates` (window omitted).

## Engine (pure, shared)

New module `src/lib/growdata/eventRules.js`:

```js
export function occurrencesForDate(date, config, eventRules) -> string[]
```

For each enabled rule:

1. Resolve the window for `date`:
   - `range`: `date` within `[from, to]` inclusive.
   - `phase`: `getPhase(date, config)` is in `rule.window.phases`.
   - `milestone`: compute `[anchorDate + offsetStart, anchorDate + offsetEnd]`; `date` within
     inclusive. Skip the rule if `anchor` is not a known config key.
   - For `cadence.type === "dates"`, the window check is skipped; the date list governs.
2. If inside the window, test the cadence:
   - `everyDay`: always true.
   - `everyNDays`: `daysBetween(anchorOrConfigStart, date) % n === 0` and not before the anchor.
   - `weekdays`: weekday of `date` is in `days`.
   - `dates`: `date` is one of `dates`.
3. If both pass, emit `rule.task`.

Return matching `task` strings ordered by `createdAt` (stable insertion order). The module is pure
(no React, no I/O) so the Cloudflare Worker imports it directly, exactly as `worker/` already
imports `getDetail` and `DEFAULT_CONFIG` from `src/lib/`.

Reuses existing date helpers from `src/lib/dates-core.js` (`daysBetween`, `sameDay`) and `getPhase`
from `src/lib/growdata/phase.js`.

## Injection point

`getDetail` gains a sixth argument `eventRules`:

```js
getDetail(date, config, overrides, generatedPlan, phaseOverrides, eventRules)
```

After the base/phase detail is assembled (base generation, then `applyPhaseOverride`) and **before**
`applyDayOverride`, append `occurrencesForDate(date, config, eventRules)` to `base.tasks`.

Why before the day override:

- By the time day overrides apply, the spray is a real indexed task, so the existing per-day
  `removedTasks` / `editedTasks` machinery skips or rewrites a single occurrence with no new code.
  "MJ, skip the spray on July 14" = a day override that removes that index.
- Every existing consumer renders recurring tasks consistently once the new argument is passed.

### Ordering and index stability

Recurring occurrences are appended after the generated/phase tasks, ordered by rule `createdAt`.
Adding a rule never reorders existing tasks (new rule sorts last). Deleting a rule shifts the
indices of later rule-tasks — the same index fragility the app already accepts for phase/day edits.
This is documented and accepted for v1. Check-offs remain index-based, unchanged.

### Call sites to update (pass `eventRules`)

`getDetail` gets a sixth parameter `eventRules = []` with a default empty array, so existing
three-to-five-argument callers (notably the test suite) keep working unchanged. Only the live
rendering paths are updated to pass the real rules:

Frontend:

- `src/App.jsx` (day view)
- `src/components/Calendar.jsx`
- `src/components/BuddyView.jsx`
- `src/lib/planText.js` (`buildPlanText` gains an `eventRules` parameter and threads it through)

Worker:

- `worker/mj/tools.js` (three call sites: get_week, get_day, set_tasks_done)
- `worker/mj/chat.js` (passes `raw.eventRules` into `buildPlanText` and `executeTool`)
- `worker/mj/undo.js`
- `worker/push.js`
- `worker/report.js` (two call sites)

Each of these already loads the raw grow, so `eventRules` is available locally once `loadRawGrow`
and `getGrow` return it and `usePlan` exposes it on the frontend.

## Persistence and API

- `worker/grows.js` `loadRawGrow` and `getGrow` parse and return `eventRules` (default `[]`);
  `getGrow` includes it in the `GET /api/grows/:id` payload that `usePlan` already consumes.
- `worker/grows.js` `ensureMigrated` CREATE TABLE adds the `event_rules` column so brand-new users
  get it; the numbered migration adds it to existing grows tables.
- `src/lib/usePlan.jsx` stores `eventRules` in context (from `data.eventRules`) and passes it into
  the four frontend `getDetail` / `buildPlanText` calls.
- New grow-scoped Worker endpoints in `worker/grows.js`, mirroring the existing `putGrowPhase` /
  `deleteGrowPhase` handlers and routed in `worker/index.js` with a regex like
  `^/api/grows/([A-Za-z0-9]+)/events(?:/([A-Za-z0-9_]+))?$`:
  - `POST /api/grows/:id/events` — create. Server assigns `id` (`evt_` + random) and `createdAt`,
    validates shape, appends to the array, persists. Returns the created rule.
  - `PATCH /api/grows/:id/events/:ruleId` — partial edit (toggle `enabled`, change `task` / `label`
    / `window` / `cadence`).
  - `DELETE /api/grows/:id/events/:ruleId` — remove by id.
- Validation lives in a pure helper `validateEventRule(rule)` in a new `worker/eventRulesValidate.js`
  (unit-tested without a DB), reusing existing constant sets:
  - `window.type` in {range, phase, milestone}; `cadence.type` in {everyDay, everyNDays, weekdays,
    dates}.
  - `window.anchor` (milestone) in `VALID_CONFIG_DATE_KEYS`.
  - `window.phases` entries in `VALID_GROW_PHASES`.
  - Date strings match `YYYY-MM-DD`; `everyNDays.n` is a positive integer; `weekdays.days` in the
    mon..sun set.
  - `task` non-empty, length cap of 200 chars; `label` cap of 80 chars; max 50 rules per grow.
    Reject over-limit / invalid shapes with `400`.
- New API wrappers in `src/lib/api.js` (`createGrowEvent`, `patchGrowEvent`, `deleteGrowEvent`) for
  parity with the existing `saveGrowPhase` / `clearGrowPhase` wrappers; not strictly required for
  MJ-only writes but kept for future UI use.

## MJ tools

Add to `MJ_TOOLS` in `worker/mj-logic.js` and implement in `worker/mj/tools.js`:

- `create_event_rule` — parameters mirror the rule model: `label`, `task`, `window` (typed object),
  `cadence` (typed object). MJ resolves natural language into the structured window+cadence. Subject
  to the standard confirmation protocol (show the grower what will be created, get explicit yes,
  then call). The tool executor validates via `validateEventRule`, writes to the active grow's
  `event_rules` column, and returns the created rule (with id) so MJ can confirm specifically.
- `delete_event_rule` — by `id`. Removes the matching rule from the active grow's `event_rules`.

Both tools operate on the active grow (`growId` in `executeTool`); if no active grow is selected
they return the same "No active grow selected" error used by the other grow-editing tools.

Surfacing existing rules:

- Extend `get_grow_info` to include the current event rules (id, label, and a short window/cadence
  summary) so MJ can reference and delete them.
- Include a one-line-per-rule summary in the season-overview context string so MJ is aware of active
  rules without a tool call.

Editing in v1 is delete + recreate through MJ. A dedicated `update_event_rule` tool is a
fast-follow, not part of v1 (the `PATCH` endpoint already exists for it).

## Migration

- `migrations/004_event_rules.sql`: `ALTER TABLE grows ADD COLUMN event_rules TEXT;`
- Add `event_rules TEXT` to the `grows` CREATE TABLE in both `schema.sql` and the
  `ensureMigrated` DDL in `worker/grows.js`, so fresh environments and newly-migrated users get the
  column.
- No backfill: a NULL column reads as an empty rule list.

## Testing

- Unit tests for `occurrencesForDate` covering each window type, each cadence type, the four
  combined patterns, boundary dates (inclusive endpoints), negative milestone offsets, disabled
  rules, and `everyNDays` anchoring.
- A `getDetail` test asserting injected occurrences land after phase tasks and before day-override
  edits, and that a `removedTasks` override correctly skips a single occurrence.
- Server validation tests for rejected shapes (bad enum, unknown anchor/phase, over-limit count).

## Out of scope for v1

- Month-calendar markers for event days (grower chose task-line-only).
- A dedicated structured editor screen (grower chose MJ-driven).
- Per-occurrence editing beyond the existing skip mechanism.
- `update_event_rule` MJ tool (delete + recreate covers v1).

Because the rule store is structured, each of these can be added later without reworking the model.
